# Shelter Finder v0.6 ‚Äî FR-Alert, OSRM, Drill Mode, TTS

## Overview

Four new features for Shelter Finder v0.6:

1. **FR-Alert / SAIP integration** ‚Äî auto-trigger alerts from French government sources
2. **OSRM real routing** ‚Äî replace haversine with actual walking distances
3. **Drill mode** ‚Äî simulate alerts without real notifications
4. **TTS announcements** ‚Äî voice alerts on speakers

All four are independent. They share an expanded OptionsFlow (multi-step).

---

## Feature 1: FR-Alert / SAIP Integration

### Goal

Automatically trigger Shelter Finder alerts when French government alert systems detect a threat near `zone.home`.

### Architecture

A provider-based system. Each provider polls one data source. An `AlertProviderManager` orchestrates active providers.

```
AlertProviderManager
  ‚îú‚îÄ‚îÄ GeorisquesProvider (API Georisques ‚Äî floods, earthquakes, industrial risks)
  ‚îú‚îÄ‚îÄ MeteoFranceProvider (vigilance meteo ‚Äî storms, snow, heat)
  ‚îî‚îÄ‚îÄ (future: SaipFeedProvider when SAIP publishes a stable feed)
```

### Provider Interface

```python
class AlertProvider(ABC):
    @abstractmethod
    async def async_fetch_alerts(self, lat: float, lon: float, radius_km: float) -> list[GouvAlert]

@dataclass
class GouvAlert:
    alert_id: str
    threat_type: str        # mapped to THREAT_TYPES
    severity: str           # "minor", "moderate", "severe", "extreme"
    title: str
    message: str
    source: str             # "georisques", "meteo_france", "saip"
    zone_lat: float
    zone_lon: float
    starts_at: datetime
    expires_at: datetime | None
```

### Threat Mapping

| Source | Event | Shelter Finder threat_type |
|--------|-------|---------------------------|
| Meteo France vigilance orange/rouge vent | Vent violent | `storm` |
| Meteo France vigilance orages | Orages | `storm` |
| Meteo France vigilance inondation | Pluie-inondation | `flood` |
| Georisques seisme | Seisme | `earthquake` |
| Georisques industriel | ICPE / Seveso | `nuclear_chemical` |
| Georisques inondation | Crue | `flood` |

### AlertProviderManager

- Polls active providers every 60s (configurable)
- Filters alerts: is the alert zone within `radius_km` of `zone.home`?
- If a new alert matches and no Shelter Finder alert is active ‚Üí trigger `shelter_finder.trigger_alert`
- If the alert expires or disappears from the source ‚Üí optionally auto-cancel (configurable)
- Deduplication by `alert_id` to avoid re-triggering the same alert
- Stores active alert IDs in memory

### Configuration (Options page "Sources")

- Checkboxes for each provider: Georisques, Meteo France
- Polling interval (default 60s, min 30s, max 300s)
- Alert radius in km (default: same as shelter search radius)
- Auto-cancel when alert expires (default: true)
- Minimum severity to trigger (default: "severe")

### API Details

**Georisques:** `https://georisques.gouv.fr/api/v1/` ‚Äî endpoints `/gaspar/risques` and `/installations_classees`. JSON REST. No auth required. Rate limit ~1 req/s.

**Meteo France:** `https://vigilance.meteofrance.fr/data/vigilance.json` or `https://public-api.meteofrance.fr/public/DPVigilance/v1/` (API key required for v1, but vigilance.json is public). Use the public JSON feed first, upgrade to API v1 later if needed.

## Feature 2: OSRM Real Routing

### Goal

Replace haversine (straight line) distance/ETA with actual walking route distance and duration using OSRM.

### Architecture

A `RoutingService` class that wraps OSRM API calls with automatic fallback to haversine.

### RoutingService API

```python
@dataclass
class RouteResult:
    distance_m: float
    eta_seconds: float
    source: str  # "osrm" or "haversine"

class RoutingService:
    async def async_get_route(self, lat1, lon1, lat2, lon2, mode="foot") -> RouteResult
```

### OSRM API Call

```
GET /route/v1/foot/{lon1},{lat1};{lon2},{lat2}?overview=false
```
Response: `{"routes": [{"distance": 1234.5, "duration": 890.2}]}`

Note: OSRM uses `lon,lat` order (not `lat,lon`).

### Caching

In-memory LRU cache keyed on `(round(lat1,4), round(lon1,4), round(lat2,4), round(lon2,4))`. TTL 5 minutes. Max 500 entries. This avoids re-querying the same routes on every sensor poll (30s default).

### Fallback

If OSRM call fails (timeout 5s, HTTP error, connection refused):
1. Log a warning (once per 10 min to avoid log spam)
2. Return haversine result with `source="haversine"`
3. Do NOT mark sensor as unavailable

### Configuration (Options page "Routage")

- Checkbox: "Activer le routage OSRM" (default: off)
- Radio: "Serveur public" / "Serveur personnel"
- URL field if "Serveur personnel" selected (default: `https://router.project-osrm.org`)
- Transport mode: walking (default) / driving

### Integration Points

- `sensor.py` ‚Äî `_find_nearest_shelter()` calls `routing_service.async_get_route()` instead of `haversine_distance()` for each shelter candidate
- `alert_coordinator.py` ‚Äî `get_best_shelter()` uses routing service for more accurate ranking
- `shelter_logic.py` ‚Äî `rank_shelters()` accepts optional route distances

### Performance

With 48 shelters and 2 persons, that's 96 OSRM calls if cache is cold. At ~50ms each, ~5s total. With cache warm (most polls), 0 calls. Acceptable.

For large shelter counts (100+), batch the top 10 nearest by haversine first, then OSRM only those. This keeps OSRM calls bounded.

- Modify: `custom_components/shelter_finder/sensor.py` ‚Äî use `RoutingService`

## Feature 3: Drill Mode

### Goal

Simulate any alert scenario for practice without sending real push notifications at high priority.

### Trigger

Existing service `shelter_finder.trigger_alert` gains an optional `drill` parameter:

```yaml
service: shelter_finder.trigger_alert
data:
  threat_type: storm
  drill: true          # optional, default false
  message: "Exercice mensuel"
```

### Behavior Differences

| Aspect | Real Alert | Drill |
|--------|-----------|-------|
| Shelter scoring | Normal | Normal (same threat_type scoring) |
| Map banner | Red `#e53e3e`, "ALERTE: {TYPE}" | Yellow `#d69e2e`, "EXERCICE: {TYPE}" |
| Push notification title | "Shelter Finder - {type}" | "[EXERCICE] Shelter Finder - {type}" |
| Push priority | high | normal |
| TTS prefix | (none) | "Ceci est un exercice." |
| Sensors | Update normally | Update normally |
| `binary_sensor.alert` state | on | on |
| `binary_sensor.alert` attribute `drill` | false | true |

### Implementation

- `AlertCoordinator.trigger(threat_type, triggered_by, drill=False)` ‚Äî stores `_is_drill`
- `AlertCoordinator.is_drill` property exposed
- `binary_sensor.py` ‚Äî add `drill` to `extra_state_attributes`
- `__init__.py` ‚Äî service schema adds `vol.Optional("drill", default=False): cv.boolean`
- `__init__.py` ‚Äî `_send_alert_notifications` checks `alert_coordinator.is_drill` to adjust title/priority
- `shelter-map-card.js` ‚Äî reads `drill` attribute, sets banner color/text accordingly
- `button.py` ‚Äî add a third button `ShelterDrillButton` with `drill=True`

## Feature 4: TTS Announcements

### Goal

Announce alerts on speakers in French when an alert triggers (real or drill).

### Message Templates

**Real alert:**
```
Alerte {type_fr}. Dirigez-vous vers {shelter_name}, a {distance} metres, environ {eta} minutes a pied.
```

**Drill:**
```
Ceci est un exercice. Alerte {type_fr}. Dirigez-vous vers {shelter_name}, a {distance} metres, environ {eta} minutes a pied.
```

**Threat type French labels:**
| threat_type | French |
|-------------|--------|
| storm | tempete |
| earthquake | seisme |
| attack | attaque |
| armed_conflict | conflit arme |
| flood | inondation |
| nuclear_chemical | nucleaire chimique |

### TTSService

```python
class TTSService:
    async def async_announce(self, threat_type, shelters_by_person, is_drill=False)
```

Flow:
1. Build message string for the closest person's shelter (or one message per person if multiple speakers)
2. Get target media_players (configured list, or all that are on/idle)
3. Save current volume of each speaker
4. Set volume to alert level (default 80%)
5. Call `tts.speak` service with the message on each target
6. Wait for estimated message duration + 2s buffer
7. Restore volume

### Auto-detection of TTS Service

1. Check if user configured a specific TTS service in options
2. If not, scan `hass.services.async_services()` for `tts` domain
3. Use first available: `tts.google_translate_say`, `tts.cloud_say`, `tts.speak`
4. If none found, log warning and skip TTS

### Configuration (Options page "Notifications")

- Checkbox: "Activer les annonces vocales" (default: off)
- Multi-select: media_player entities (empty = all available)
- Select: TTS service (auto-detect / manual selection)
- Slider: volume d'alerte (0-100%, default 80%)

## OptionsFlow Multi-Step

### Current State

Single page with: radius, travel mode, overpass URL, cache TTL, adaptive radius, re-notification settings.

### New Structure

| Step | Page | Fields |
|------|------|--------|
| 1 | Sources & Rayon | search_radius, adaptive_radius, overpass_url, cache_ttl, **provider checkboxes** (Georisques, Meteo France), polling_interval, min_severity, auto_cancel |
| 2 | Routage | osrm_enabled, osrm_mode (public/self-hosted), osrm_url, transport_mode |
| 3 | Notifications | re_notification_interval, max_re_notifications, **tts_enabled**, tts_service, tts_media_players, tts_volume |
| 4 | Avance | custom_osm_tags, overpass_url |

Each step calls `self.async_show_form(step_id=...)` and returns the next step or `self.async_create_entry()`.
